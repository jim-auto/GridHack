# ゲームプレイ・アーキテクチャ

## 現在のプロトタイプ構成

- エンジン: Godot 4.3+
- 言語: GDScript
- 視点: トップダウン 2D
- シミュレーション方針: グリッド管理の地形と伝播、プレイヤー移動は連続移動
- 現在のモード: 人間1人 + ボット1体のローカルFFAプロトタイプ

## 実行時レイヤー

### `Main`

マッチセッションを管理します。

- セクターの起動とリセット。
- プレイヤー生成。
- アクティブな感染ノードをセル単位で登録。
- 汚染解決をプレイヤー、ノード、ファイアウォールへ振り分け。

### `ArenaGrid`

ネットワークトポロジーを管理します。

- プロシージャルなセクター生成。
- タイル状態: 空き、ハードファイアウォール、ソフトファイアウォール、不安定セクター、ゲート。
- 一時ファイアウォール。
- データストリーム。
- 伝播用のネットワーク隣接セル検索。

### `PlayerAI`

プレイヤーまたはアバターの挙動を管理します。

- グリッド制約付きの連続移動。
- 感染ノード設置。
- 一時ファイアウォール生成。
- インテグリティ、撃破、再起動ループ。
- 圧力テスト用の簡易ボット挙動。

### `InfectionNode`

遅延型の空間制圧脅威を管理します。

- 時間経過でアーム。
- 汚染によって連鎖起動可能。
- 起動時に伝播リゾルバを呼び出す。
- 起爆後、所有者の設置上限を解放。

### `PropagationResolver`

汚染の拡散を管理します。

- ネットワーク隣接セルをBFSで走査。
- パケット移動として読めるように、セル解決に遅延を持たせる。
- 視覚パルスを描画。
- 連鎖起動とダメージ解決は `Main` 経由で行う。

### `Hud`

マッチ中に読むべきUIを管理します。

- プレイヤーのインテグリティ。
- 感染ノードの再使用状態。
- ファイアウォールの再使用状態。
- セクターシードとモード表示。

## データモデル

セルは `Vector2i` 座標です。ワールド座標は `ArenaGrid.cell_to_world()` から取得します。

主要な辞書:

- `Main.active_nodes[cell] -> InfectionNode`
- `ArenaGrid.temp_firewalls[cell] -> remaining_seconds`
- `ArenaGrid.streams[from_cell] -> to_cell`
- `ArenaGrid.gateways[cell] -> paired_cell`

## グラフ伝播を採用する理由

このプロトタイプでは、半径爆発や十字爆発ではなく、`ArenaGrid.get_network_neighbors()` を通じたBFS伝播を使います。これにより、同じ感染システムで次を扱えます。

- 通常のネットワーク回廊。
- 一方向のデータストリーム。
- ゲートによる跳躍。
- 不安定セクターの斜め分岐。
- 将来のハック経路やチーム所有インフラ。

## 次のアーキテクチャ作業

- プレイヤー入力をコマンドオブジェクト化し、決定論的リプレイとロールバックに備える。
- ネットワーク対応開始時に、直接の `game` 参照をマッチイベントへ置き換える。
- プロトコル拡張を、プレイヤー、ノード、ファイアウォール挙動への小さな修飾コンポーネントとして追加する。
- 伝播プレビューを調整できるデザイナー向けパラメータUIを追加する。
