# マルチプレイヤー構成案

## 目標

2〜6人のアリーナマッチを対象に、決定論的な結果、低遅延の移動、信頼できる感染連鎖解決を実現します。

## 推奨モデル

最初は、権威ホスト/サーバー + 決定論的コマンドシミュレーションで開始します。

理由:

- 完全なピアツーピア・ロールバックより構築が速い。
- チート対策がしやすい。
- 伝播と連鎖反応の再現性を保てる。

## シミュレーション規則

- 固定シミュレーションtick。推奨は30Hzまたは60Hz。
- クライアントは座標ではなく入力コマンドを送る。
- サーバーが受理されたコマンド順を所有する。
- 感染設置、伝播走査、ダメージはtick上で解決する。
- ランダム生成はマッチシードと決定論的ストリームIDを使う。

## コマンド形式

各プレイヤー、各tick:

- 2軸に量子化された移動ベクトル。
- 感染ノード設置ボタン。
- ファイアウォール生成ボタン。
- 任意のプロトコル拡張コマンド。
- クライアント入力シーケンス番号。

## 状態同期

高頻度:

- プレイヤー位置。
- プレイヤー速度または移動意図。
- インテグリティ。

イベントストリーム:

- 感染ノード設置。
- 感染ノードのアーム/起動。
- 伝播パルス発生。
- ファイアウォール生成/消滅。
- ゲート転送。
- プレイヤー再起動。

低頻度:

- セクター全体のチェックサム。
- アクティブノード一覧。
- 一時ファイアウォールのタイマー。

## 決定論の優先順位

まず決定論化するもの:

- アリーナ生成。
- 伝播BFS順序。
- ゲートとストリーム挙動。
- 感染タイマー。
- ダメージタイミング。

補間または見た目だけでよいもの:

- グリッチ粒子。
- HUDアニメーション。
- パルス残光。
- カメラシェイク。

## ロールバックへの移行手順

権威方式のプロトタイプが安定した後:

1. N tickごとにコンパクトなスナップショットを保存する。
2. コマンド履歴を保持する。
3. ローカル移動にクライアント予測を追加する。
4. サーバー補正でリコンシリエーションする。
5. 最初はプレイヤー移動と感染設置だけをロールバックする。
6. 連鎖反応バグを把握してから、伝播ロールバックを追加する。

## アンチチート

- サーバーが現在のグリッド状態に対してノード設置を検証する。
- サーバーが一時ファイアウォール設置を検証する。
- サーバーが汚染ヒットを所有する。
- クライアントはダメージや連鎖起動を宣言できない。
- プロトコル拡張はサーバー管理のデータにする。

## Godot 実装メモ

- 実ネットワーク対応前に、マッチ状態を純粋なシミュレーションクラスへ移す。
- レンダリングノードはシミュレーションイベントの購読者にする。
- マッチクリティカルな処理では浮動小数タイマーではなく整数tickを使う。
- オンライン化前にリプレイ保存/読み込みを追加する。決定論バグの発見が早くなる。
